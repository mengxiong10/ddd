# AGENTS

开发前必读 [CONSTITUTION.md](CONSTITUTION.md)（项目总纲）与 [CONTEXT.md](CONTEXT.md)（术语，若已存在）。
所有开发按 `specs/NN-<feature>/` 下的 `prd.md` 与 `dev.md` 约束进行，不要绕过文档直接堆代码。

## 架构红线

- 依赖方向单向：对局推进为 `ui → store → core`；开局数据为 `store → data/scenarios → core`。core 只推进已初始化的 `GameState`，不持有、枚举或加载具体剧本。
- `src/core/` 为纯 TS，禁止 import React / Zustand / 任何 UI。
- 游戏规则只写在 `core`；`store` 只做状态/存档；`ui` 只渲染与派发 action。
- `core` 内部按限界上下文组织（world / economy / military / turn / ai / shared），采用 **DDD-lite（函数式）**：聚合=数据类型+纯函数、值对象、领域服务、统一语言。
- 不引入仓储/应用层/事件总线/实体 class；YAGNI，出现真实需求前不提前抽象。
- 跨上下文协调只走 `core/game.ts` 的 `apply` 总入口委派；上下文之间不互相深耦合。
- 指令分「校验/变更」两段：`canX` 返回统一的 `CommandCheck`（`shared/command.ts`，`{ok, reason}`），`X` 执行变更且非法即 no-op；`game.canApply/apply` 按 Action 分派委派给各领域服务。
- **经营命令三阶段同处注册**（`economy/registry.ts`）：每条 economy 命令把 `can`（校验，薄适配 `canX`）/`call`（下令执行，薄适配 `X`）/`run`（月末执行，薄适配 `executeX`，仅延后效果命令有）注册在 `SPECS` 同一条目下，**key 即 action type**。`game.canApply/applyWithEvents` 经 `isEconomyAction` 收窄后查 `economyCan/economyCall` 分派、其余为阶段动作（battle/resumeMonth/chooseSuccessor/chooseDefenders/endMonth）保留手写分派；`turn/pending.runNonCampaignPending` 查 `economyMonthRun` 表分派 `run`、缺省月末空操作。命令的 `canX/X/executeX` 实现签名不变，registry 仅薄适配。**Action type 与 PendingCommand type 一一对应**（开垦/招商即 `reclaim`/`commerce` 两个独立 type，不再合用 `develop`），故入队的队列项 type 直接取 `SPECS` 的 key，不另存 `commandType`——`economyMonthRun` 即按 key 收集声明了 `run` 的条目。
- 占人指令只传 `officerId`（不传 `cityId`）：作用城 = `officer.cityId`（单一真相源）。`scout` 例外，额外带真输入 `targetCityId`（被侦察的敌城）。
- 归属（只能指挥自己的将）不在 `core` 校验：`apply` 保持极简、actor-agnostic。行动方身份属入口的环境约束——玩家走 `store` 派发口（注入 `playerLordId`，校验 `officer.lordId === state.playerLordId`），AI 只对自己麾下武将造 action。`lordId` 不作为 Action 字段（单机无信任边界，传了也可伪造）。
- **所有占人指令均入队** `GameState.pendingCommands`（带 `type` 的判别式并集），由 `turn` 层经 `economy/registry` 的 `economyMonthRun` 表按 `type` 分派执行（与 `game.apply` 经同表 `can/call` 分派同构）：有月末效果者执行其 `run`（掠夺/移动/输送/搜寻/招降/外交）；**即时生效指令**（开垦/招商/出巡/治理/交易/侦察/征兵）效果已于下令时结算，无 `run`、月末为空操作、仅作占用标记。**占用中为派生**（`world/queries.isBusy` = 被某条 pending command 引用），不存 `Officer.busy`——**占人=入队、释放=出队**（详见下方"占用为派生"红线）。月末顺序固定：`pendingCommands → settle（收粮/收税）→ 体力恢复 → 月份+1`（队列在月末清空即所有派生占用归零，无独立"回城"置位步骤）。`pendingCommands` 内部两趟执行：先所有非 `campaign`（掠夺/即时类等）按入队序，再所有 `campaign`（出征）按入队序。
- **俘虏为派生状态、不存字段**：`officer.lordId !== 所在城.lordId` 即俘虏（`world/queries.isCaptive`）。"在任武将"（`officersInCity(onlyAvailable)`）排除俘虏；占领只改 `city.lordId`，守军原地自动成俘虏。**占人例外（武将月末改变位置）**：出征——胜方进驻目标城、败方按战后命运被俘/逃跑/战死（`cityId` 由 `military/aftermath.resolveCampaignOutcome` 改写）；移动——月末落到目标己方城（`cityId` 由 `executeMove` 改写）。两者位置均由月末执行改写、turn 层无特例。`pendingCommands` 非 campaign 趟含掠夺/移动/输送，campaign 趟含出征。
- **城邻接拓扑放 `GameState.adjacency`**（值对象，fixture 播种）：非全局常量、不进 config，使 `apply/canApply` 签名不变、可注入测试、随存档序列化。
- **道具加成只经 `queries.effectiveOfficer` 收敛**：道具对武力/智力的加成是派生，不写回 `Officer` 存储字段。所有用到武力/智力的公式（带兵量、开垦/招商增量、掠夺产出、重选君主智力比较）输入一律取 `effectiveOfficer(state, officerId)`，`troopCapacity(o)` 等纯公式签名不变。新增此类公式时同样吃有效值。
- **武将忠诚派生君主恒 100**：`Officer.loyalty` 为存储字段，但对外读取走 `queries.officerLoyalty`——君主（`officer.lordId===officer.id`）恒返回 100（即便经重选君主换人也成立），赏赐/没收对君主跳过写入。
- **即时·不占人指令**：赏赐/没收（及分配）效果在下令瞬间结算，**不入 `pendingCommands`（故 `isBusy` 派生为假、不占用）、不耗 RNG**，且**不校验占用**（君主对武将下令，武将本月仍可被其他指令占用）。归属（己方）校验同样不在 `core`，留 store 派发口。
- **登场/在野/未发现（`06-debut-search`，`20-original-scenarios` 统一模型）**：本剧本的现存与未来武将/道具从开局起均在 `GameState.officers/items`，不设独立待登场池。未登场由位置为空派生：`Officer.cityId=null`、独立待登场道具 `Item.holder=null`；二者必带 `appearanceConditions={birth,recruiterId,cityId}`。`world/debut.runDebuts` 在月末「月份+1」后按武将数字 id 升序、再按独立道具数字 id 升序处理，武将判 `year≥birth+16`、道具判 `year≥birth`，指定城为空时随机选城并消费 RNG。`Officer.lordId=null` 仍表示无主；只有已登场且位于城市的无主武将才是在野候选。未登场实体不参与城市、指令、战斗或搜寻。
- **未登场武将装备直接持有（`20-original-scenarios`）**：装备从开局起进入 `GameState.items`，holder 直接指向未登场武将并保留 `equipSeq`、`discovered=true`；道具随持有者不可操作，武将登场只填 `cityId`，装备自然生效，不做二次物化。
- **性格单值双表解读**：`Officer.personality`（`0..4`）单存一处；君主表（和平/大义/奸诈/狂人/冒进）/普通武将表（忠义/大志/贪财/怕死/卤莽）由 `lordId===id` 派生切换、不另存第二份；重选君主后新君主自动改用君主表解读其原值。文字标签属 UI，不入 core。
- **无己方执行人的处置类指令归属口径**：处斩/流放（目标可能是敌方俘虏，无己方执行人）按**作用城归属**（`city.lordId===playerLordId`）在 store 派发口校验，区别于占人指令按执行人归属（`officer.lordId===playerLordId`）；`core` 仍 actor-agnostic、不校验归属。
- **下令 vs 战斗结算分上下文**：所有指令的**下令阶段**（`canX`/`X`：校验、扣本城资源、占人、入队）归 `economy/`，形态一致、与指令面板对应；**战斗的战中/战后结算**（`military/battle` + `military/aftermath`：战斗、占领、败军命运、君主遭劫）归 `military/`。两半经 `game.apply`、`turn/pending` 两个既有分派点接线，互不 import。经营领域的月末执行（掠夺破坏本城、收粮收税）仍就近留 `economy/`。
- **城状态自治月末事件归 `world/`（`09-city-disaster`）**：灾害破坏/生成/恢复**不挂在任何玩家指令上**、只读写 `City` 的状态/防灾值/资源字段，故归 `world/disaster.runDisasters`，区别于「挂在指令延后效果上的月末执行（掠夺破坏/收粮收税）就近留 `economy/`」。判据：月末某段逻辑是否是某条玩家指令的延后效果——是→economy，否（城状态自治）→world。`City.status`（`'normal'|'famine'|'drought'|'flood'|'riot'`）与 `disasterPrevention` 为不可派生的城自治真相、必存；破坏的逐字段变换（`applyDisasterDamage`）与 `ravage` 同位收敛在 `world/city.ts`，`disaster.ts` 只管 RNG 判定与按城（id 升序）遍历编排。月末固定顺序**纯追加**为 `pendingCommands → settle → 体力恢复 → 月份+1 → 登场 → 灾害（runDisasters）`，不挪动既有步骤、不改 6/10 收粮与 3/6/9/12 收税日历。治理指令的下令阶段照旧归 `economy/govern`。
- **外交指令归经营·月末（`10-diplomacy`）**：招揽/离间/策反/劝降四条 `占人 ✓ · 效果=月末`，下令阶段与月末执行（含 territorial `lordId`/`cityId` 改写）就近收敛于 `economy/diplomacy.ts`，沿用招降在 economy 改归属的先例，**不**触 `military`/`world/succession`。目标用 `targetOfficerId`（敌方武将/太守/君主），`PendingCommand`/`Action` 四分支同形。招揽/离间/策反共用三关（智力差→忠诚→性格，内部 `runThreeGates` 去重）、劝降另套关（城池压制+智力差+君主性格、无忠诚关）。
- **太守为派生（`world/queries.governorOf`，`10-diplomacy`）**：某城太守 = 君主正驻该城则为君主、否则本城在任武将中有效智力最高者（平局取 id 最小）；零存储字段。策反目标须为**非君主太守** → 君主即太守时不可策反 → 分裂城永不含原君主 → **策反不触发重选君主**（结构上免除接 succession）。
- **core actor-agnostic 的显式例外（`10-diplomacy`）**：劝降「玩家君主免疫」是**游戏规则**（非归属校验），`executeInduce` 显式读 `state.playerLordId`、目标为玩家君主即失败（防 AI 劝降玩家君主）。除此唯一例外外，`core` 不读 `playerLordId`、不校验归属。
- **兵种=基础存储 + 有效派生（`11-troop-types`，`20-original-scenarios` 扩展）**：`Officer.troopType` 只存**基础兵种**（fixture 可信播种、不跑 >105 门槛）；**有效兵种**与**移动力**经 `queries.effectiveTroopType`/`officerMovement` 派生、**不写回 Officer**，与 `effectiveOfficer`（有效武力/智力）同构。有效兵种 = 基础兵种被所持「改兵种」道具按装备先后依次覆盖（后者覆盖前者）、门槛（玄兵智力>105/极兵武力>105）取 `effectiveOfficer` 的有效值实时判定（故没收道具会回退）。改兵种字段 `0..6`：`0` 不改、`1` 水军、`2` 玄兵、`3` 极兵、`4` 骑兵、`5` 步兵、`6` 弓兵；仅玄/极有属性门槛。兵种规则收敛在纯模块 `world/troop-type.ts`（移动力表/门槛常量/`resolveOverride`）。下游（战斗）需要时**临场读** `effectiveTroopType`，不预埋。
- **原版剧本稳定数字身份与共享目录（`20-original-scenarios`）**：原版快照只由 `scripts/scenarios/generate-original.mjs` 解释；生成产物拆为共享 `cities/officers/items/adjacency` 与四份时期状态，放在 `src/data/scenarios/generated/` 并由 `src/data/scenarios/index.ts` 合并、创建初始 `GameState`。core 不枚举或加载具体剧本。生成 JSON 禁止手改，源数据或转换规则变化后必须重跑并通过 `--check`。`CityId/OfficerId/ItemId` 均为从 1 开始的普通 number、各自独立编号空间；城/道具沿用原版编号，武将跨时期使用首次出现分配的稳定目录 ID。姓名校订只在生成器显式执行，当前为 `荀或→荀彧`、`李决→李傕`、`博士仁→傅士仁`、`张合→张郃`，不得改动既有数字 ID。所有按 id 的遍历、RNG 消费与平局规则采用数字升序。单剧本同一道具至多一件：生成器按 `goods_queue`、再按武将原始编号/装备槽扫描，只派发首个引用。
- **原版共享地图数据（`20-original-scenarios` 扩展）**：`City.x/y` 是跨剧本固定的 `12×9` 世界网格坐标，只服务布局与开战时派生攻击方向，不派生邻接；出征可达性仍唯一读取 `GameState.adjacency`。`City.battleMapId` 引用稳定数字 `BattleMapId 1..7`，同城跨剧本固定。仓库只保留裁剪后的七张 `32×32` 原版纯地形输入（不含 tile/图片/资源号），生成期 `hill→mountain` 后与其它产物一同写入 `src/data/scenarios/generated/`，由 `createScenarioState` 构造 `GameState.battleMaps` 注入 core；core 不 import 具体地图。`BattleMap` 只存地形，唯一 `city` 格直接作为胜负点；攻方出生点按出发城相对目标城的八方向、守方出生点按城池格，在开战时依原版 `dFgtIntPos` 动态计算；禁止 `plains` 生产占位或未知 id 静默回退。
- **空城是可占领领土（`20-original-scenarios`）**：`City.lordId=null` 表示未归属，不造中立君主；空城跳过 AI 与灾害且不消费对应 RNG，仍按既有规则收粮/收税。移动仍只允许己方城，空城只能经出征占领。
- **装备顺序经 `holder.equipSeq`、不拆归属模型（`11-troop-types`）**：保留 `ItemHolder` 判别式（属城 XOR 属将的结构安全不变），officer 分支加 `equipSeq` 表达装备先后；`itemsOfOfficer` 按 `equipSeq` 升序返回。**唯一**写 officer-holder 的路径是 `economy/reward`，由它算 `nextEquipSeq = 1 + 该将现有最大 seq`（首件 0）。不新增「装备/卸下」指令——沿用赏赐/没收为装备/卸下事件。
- **战斗为 `apply` 之上的交互式子对局（`12-battle`）**：玩家参与的出征不再月末同步速算，而是把 `endMonth` 改为**可重入**——遇玩家 campaign 时挂起为 `GameState.activeBattle`（非空=月末停在战斗中）、经离散 `BattleAction` 逐步推进、分胜负后由 `turn.resumeMonth` 写回（每单位兵力/经验/等级 + 复用 04 抽出的 `military.resolveCampaignOutcome` 做占城/俘虏/重选君主）并续跑剩余 campaign/月末尾段。剩余 campaign 队列即「续战清单」，无额外续点标记。战斗专属 action 经 `{type:'battle',action}` **单包装**委派 `military/battle`，`{type:'resumeMonth'}` 委派 `turn`；**`military` 不反向 import `turn`**（battle reducer 只推进战斗、置 `outcome`，写回+续月末归 turn 调 military）。**确定性红线已收窄（`13-battle-skills`）**：仅「无技能的 12-battle 子集」确定性、不耗对局 RNG；启用技能后战斗**消耗 `GameState.rng`**（技能成功率、天气逐日刷新、人物状态每日判定），`reduceBattle`/日循环读取并推进 `state.rng`、整局同 seed 可复现。对手方本切片仍不行动（仅「玩家进攻」可达，防守/AI 施法结构预留）。
- **战斗技能（计谋）为 `military/battle` 的纯规则增量（`13-battle-skills`）**：MP 仅活在 `BattleUnit` 快照、开战由 `floor((floor(智力×80/100)+floor(sqrt(武力)/2)+等级)×体力/100)` 派生（吃有效武力/智力）、**战后不写回 Officer**（本作不建 HP 模型，扣的仍是兵力）。施法是 `act` 的第三种终结动作 `cast`（与 attack/rest 同级、结束本日），混乱/石阵单位不可操作、禁咒单位不可选 cast。可用性四关=技能在「天气/施法者地形/目标地形/目标兵种」对应倍率**均≠0**（`0`=不可用），效果走 5 步逐 `floor` 倍率链。技能解锁 `floor(默认数×等级/21)+1`（默认按兵种、按列表序封顶）；个人技能=`Officer` fixture、君主技能=派生（君主额外得谍报30），二者不受解锁公式限制。纳入 27 技（去 `21遁甲/26潜踪/28急行`）；人物状态建模 `混乱/禁咒/定身/奇门/石阵/死亡(=击溃)`，每天开头刷天气→状态判定（`RandInt(0,59)<floor(智力/2)`，石阵先损兵 1/8）两步**纯插入**日循环、不动 12 既有顺序。单位在 `BattleUnit` 快照中演进、`concludeBattle` 才写回 `Officer`（不在战中污染持久态）。`死亡` 为 `status` 的一支、是击溃唯一真相（去掉旧 `routed` 字段）。技能纯规则新增三叶模块——`military/battle-weather`（天气类型/维序/刷新）+ `military/battle-status`（可控状态/谓词/石阵损兵/每日判定）+ `military/battle-skill`（技能表/9×9 掩码/默认·君主技能/MP·解锁·倍率链·四关·成功率公式），与 12 的 `battle-map`/`battle-combat`/`battle-movement` 同级，`military/battle` 仍只编排（`startDay` 统一开局/每日开头：刷天气→状态判定+石阵损兵→重置行动→胜负，第 1 天经 `turn` 调用）。`Officer.personalSkills` 用 `number[]` 而非 `SkillId` 以避免 `world→military` 反向依赖。
- **完整战后处理收敛 `military/aftermath.resolveCampaignOutcome`（`14-campaign-aftermath`）**：12 留下的极简战后逻辑深化为 6 步——①胜方存活单位进驻目标城 ②`attackerWins` 占城（仍 keep 模型，只翻 `city.lordId`）③败军逐人命运（`processDefeatedArmy`）④遭劫君主处置（`resolveStrickenLord`）⑤目标城 `applyBattleDamage`（无条件战损：农/商/金 floor×0.95、民忠 floor×0.90，不碰粮/后备兵/人口）⑥目标城 `food=mergedFood`（覆盖式粮草合并，非累加）。**唯一调用方是 `battle.concludeBattle`**（从 `BattleState` 组装 `CampaignOutcome` 结构入参：`attackerWins`/`attackerLord`/目标城/双方参战名单/`mergedFood`，其中 `attackerLord` 与攻/守名单均由 `units`+`Officer.lordId` 派生、`BattleState` 不再单存——瘦身去掉 `provisions`/`defenderLord`/`officerIds`/`attackerLord`）；已删不可达的 `executeCampaign` 速算死代码（`campaign.ts`→`aftermath.ts` 改名）。**占城必在败军处理之前**——故败方「逃跑」只能逃向其势力**其余存活城**（占下最后一城→无城可逃）。
- **确定性红线再收窄：`resolveCampaignOutcome` 也消耗 `GameState.rng`（`14-campaign-aftermath`）**。败军逐人命运（按数字 officerId 升序、吃有效智力）：①`RandInt(0,99)>有效智力`→被俘（进目标城、兵清零、`lordId` 不变=派生俘虏）②否则取 `citiesOfLord(其 lordId)`（按数字 id 升序）随机一座→逃跑成功（保留兵、非俘虏）③无城→逃跑失败：`RandInt(0,99)===0`→战死（道具 `discover(holdByCity(目标城))`、officer 永久删除），否则被俘。整局同 seed 仍可复现。
- **重选君主拆 actor-agnostic 纯工具 + 显式 `playerLordId` 例外（`14-campaign-aftermath`）**：`world/succession` 改为纯工具（`successionCandidates`/`pickSuccessor`/`promoteLord`/`canChooseSuccessor`，候选=该势力非俘虏非自身武将、自动选取有效智力最高平局 id 最小、`promoteLord` 把势力城+非俘虏武将归新君且新君忠诚 100、`oldLord===playerLordId` 时一并迁 `playerLordId`），**不读 `playerLordId`**；删旧 `resolveSuccession`。「AI 自动立新君 vs 玩家手动选」的分支由 `aftermath.resolveStrickenLord` 读 `state.playerLordId` 决定——这是 core actor-agnostic 的**第二处显式游戏规则例外**（同 10 劝降玩家君主免疫）。遭劫君主无城/无候选→灭亡（不立新君）；君主战死被删也能触发（守卫不依赖 `isCaptive`）。
- **玩家决策暂停态 `GameState.pendingSuccession`（`14-campaign-aftermath`，类比 `activeBattle`）**：玩家君主遭劫时 `resolveStrickenLord` 不换主、只设 `pendingSuccession={lordId}`（窄态、候选动态派生），`resumeMonth` 检测到即提前返回（月份不推进）；新 action `{type:'chooseSuccessor',officerId}` 委派 `turn.chooseSuccessor`——`promoteLord` 兑现换主→清空 `pendingSuccession`→续跑 `advanceCampaigns`（可重入：同月剩余 campaign 可再起战斗）。`endMonth`/`canApply` 在 `pendingSuccession` 非空时拒绝推进。纯同步、无 Promise。`turn` import `world/succession`（`promoteLord`/`canChooseSuccessor`）+ `military`（`concludeBattle`），`military/aftermath` 不 import `turn`。
- **AI 经营走「作弊简化下令 + 复用月末执行」（`15-ai-economy`）**：`core/ai/aiTakeTurn`（月末 `endMonth` 最前、玩家命令之前）对每座非玩家城（`lordId≠playerLordId`）按**君主性格**选内政/外交/军备一条路径，逐在任武将（按 id 升序）随机生成命令。AI 是**作弊简化路径**：固定成长（开垦/招商 +200、出巡民忠+4·人口+100、治理防灾+4）、不扣金/体力、不吃智力公式、**不走 `canX`**；AI 下令 = 立即施加固定效果（含**即时**招降/处斩，区别于玩家月末四关模型）或经 `busyEnqueue` 只 push `PendingCommand`（占用由队列派生，不存 `Officer.busy`），不复用会扣成本的 `search()/move()/entice()` 下令体。**月末结算复用现有 `executeSearch/executeMove/executeEntice/…`**（单一月末结算源）。全程消费 `GameState.rng`、城与武将均按 id 升序遍历、同 seed 可复现；AI 出征由 `16-ai-campaign` 在军备路径追加。布局仿 `military/battle`：`ai.ts` 编排 + `ai-shared.ts`（破环共享助手）+ `ai-internal/diplomacy/military.ts` 三叶；`ai/*` 单向依赖 economy/world/shared，不被反向依赖。
- **战斗 AI 为 `military/battle` 三拆 + 决策/结算分离（`17-battle-ai`）**：`military/battle` 拆为**编排 `battle.ts` + 核心 `battle-core.ts` + 对手决策 `battle-ai.ts`**，依赖单向——`battle →{battle-core, battle-ai}`、`battle-ai → battle-core`（运行时）、`battle-movement → battle-core`（仅 `import type`），`battle-core` 不反向 import `battle`/`battle-ai`；`battle.ts` **再导出**核心类型与 `canBattle`，外部 `from './military/battle'` 零改动。`endDay` 对手方不再 no-op：`advanceDay` 先跑 `runOpponentTurn`（循环 `battle-ai.nextOpponentAction` **决策** + `battle-core.applyActResolved` **应用**，每步即时胜负）再扣粮进下一天。**决策与结算分离**——`battle-ai` 只选将/选落点/选动作（消费 rng 于技能筛与简单选技、不结算），`applyActResolved` 是从旧 `applyAct` 抽出的「已校验即应用」核心（去 player 门），`canBattle` 仍仅校验玩家方。AI 目标点纯由模式定（`attack`=玩家主将格扑主将、`defend`=城池格中心推城），**选位只算普攻预估伤害（主将视作∞）不评估技能收益**、按兵种派生攻击优先(非玄)/技能优先(玄)与简单/最佳选技、跳过 self 技能、**从不撤退**；全程消费 `GameState.rng`、确定性可复现。AI vs AI 仍走 `16` 速算（不进地图战），本切片只动对手方、不实现玩家方 AI 托管。
- **占用为派生、不存字段**：武将"本月被占用"由 `world/queries.isBusy(state, officerId)` 派生——即"被某条 `pendingCommands` 引用"（`campaign` 看 `officerIds` 成员、其余看 `officerId`），**不存 `Officer.busy`**。**原因**：占人指令一律入队后，"占用中"与"被某条 pending command 引用"是同一事实，独立存储 `busy` 会与队列形成双写、易失同步（守军判定即因此出过 bug）。据此：**占人=入队**（指令只 push command、不写占用位）、**释放=出队**（`runNonCampaignPending` 过滤非 campaign、`dropFirstCampaign` 移除 campaign、月末 `settle` 清空队列三处天然归零，不写任何"置占用为假"、不做尾段全量清空）。即时生效指令（开垦/招商/出巡/治理/交易/侦察/征兵）入各自 `type`、月末分支为空操作，仅作占用标记。`canX` 与 `officersInCity(onlyAvailable)` 一律读 `isBusy`；守军 `defendingOfficers` 因此收敛为 `officersInCity(onlyAvailable)`。与俘虏/太守/君主忠诚同属"派生不存字段"族。
- **命令反馈零中文：reason code + OutcomeEvent，事件与 state 并列不进聚合（`18-command-feedback`）**：`core` 的命令反馈两轴皆零中文运行时串——①失败走 `shared/command.ts` 的 `ReasonCode` 字符串字面量联合（替代 `CommandCheck.reason` 的中文串、跨命令复用 `officer-not-found`/`gold-insufficient`/`is-captive` 等）；②成功/月末结算/系统事件走 `shared/outcome.ts` 的 `OutcomeEvent` 判别式联合。本地化与多变体台词由 UI 据 `docs/business-command-rules.md` 映射，**core 不挑台词、不掷骰**。**事件与 state 并列、绝不进 `GameState`**：产事件的函数返回 `WithEvents<GameState>={state,events}`，orchestrator 用 `step`/`lift` 组合子逐层 thread 并拼接。瞬态反馈不污染持久聚合（省存档剔除/全等比较），代价是**事件路径上**的 producers/orchestrators 签名改 `→WithEvents`、纯步骤经 `lift` 提升。**事件不消耗 `GameState.rng`**（只构造对象、不掷骰，确定性回归逐字节一致）。**只为"携数据/决结果分支"产事件**（随机增量、月末结算结果、系统事件）；纯确认与 UI 已知量（征兵/交易/分配/赏赐量）不产，且**不加 `command-issued` 之类纯确认事件**。**下令接口统一：每个下令 X（develop/govern/patrol 及 trade/allocate/campaign/plunder/move/scout/search/banquet/behead/banish/suborn/recruit/transport/reward/confiscate/entice/alienate/instigate/induce）一律返回 `CommandResult = CommandCheck & WithEvents<GameState>`（`outcome.ts` 的 `WithCheck<S>` 泛型，配 `commandOk`/`commandFail` 二助手）**——X **自报告**内部本就计算的 `canX` 结果（失败 `commandFail(check,state)`：带出 `reason`、`state` 不变、`events:[]`；成功 `commandOk(next,[event?])`：`ok:true`）。**校验只在 X 内跑一次**（`applyWithEvents` 不再顶层重复 `canApply`），失败 `reason` 经 X→`applyWithEvents`→`apply` 自然冒泡。`canApply`/`canX` 保留供 UI 在派发前预判（置灰/提示），与执行路径解耦。富入口 `applyWithEvents→CommandResult`：经营动作直接转发 X 的自报告结果；5 个**阶段动作**（battle/resumeMonth/endMonth/chooseSuccessor/chooseDefenders 自身无 canX 自报告改造）合并 `canApply` 取 `ok/reason`（与其内部守卫一次轻量重复，按点击节奏可忽略）。`apply=applyWithEvents(...).state`（退化包装、丢弃 ok/reason/事件，行为逐字节不变）。月末执行类（executeX/settle/debut/aiTakeTurn/succession 工具）非下令、保持各自既有签名（`WithEvents` 或 `state→state`）。**core 仍 actor-agnostic**：事件无差别产出（含 AI 经执行器入队的月末命令），"是否玩家可见"的过滤是 UI 的事。`succession-pending` 在 `military/aftermath.resolveStrickenLord` 设 `pendingSuccession` 处就近 emit、`lord-succeeded` 在 AI 自动换主与玩家 `chooseSuccessor` 两处 emit。per-unit 战后明细与战斗逐日提示不在本切片。

- **上层接入红线（`19-store-ui`）**：依赖链收成严格两段 `ui → store → core`——**UI 只 import `store`、绝不直接 import `core`**；`src/store/selectors.ts` 是 UI↔core 的唯一通道（再导出 core 查询与类型，store→core 向下合法）。**store 零规则、零中文运行时串**：只持 `GameState`+`GameConfig`+**瞬态反馈队列**（带自增 id 的 `FeedbackItem`，非 `GameState`、不持久化），`dispatch` 走 `applyWithEvents`（成功更新 game+逐条入队 events、失败入队 reason，恒返回 `CommandResult`），`canDispatch=canApply`。**中文文案唯一落点 = UI 映射模块**（`OutcomeEvent`/`ReasonCode`→中文 + 多变体台词随机 + **可见性过滤**：非玩家相关事件不展示），对照 `docs/business-command-rules.md`。store 用 zustand vanilla `createStore`（无头、node 可单测）+ `useGameStore` 钩子。
