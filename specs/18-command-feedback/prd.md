# command-feedback PRD

## 问题

`docs/business-command-rules.md` 描述了游戏面向玩家应出现的全部提示（下令成功、月末结算、系统事件、失败原因、台词）。但当前 `core` 只在 `canX` 失败时返回 `CommandCheck.reason`（一段**中文串**），命令执行与月末结算**完全不产出任何结果反馈**。结果是：

- core 里混入了中文（违背"文字标签属 UI，不入 core"红线）。
- 未来 UI 无从知道"开垦后农业变成多少、增量几何""掠得多少金粮""招降成功还是失败""哪座城出了灾害"——这些数值/结果只活在 core 内部、没有出口。

## 目标

让 `core` 为经营/外交命令与月末系统事件产出**结构化、零中文**的结果反馈：失败用 code、成功/结算/系统事件用判别式事件（带数值/id）。中文呈现（含多条随机台词的挑选）全部归 UI，以 `docs/business-command-rules.md` 为权威对照表。一句话：**core 只吐 code 与结构化事件，不吐中文。**

## 关键行为 / 用户故事

1. 作为开发者，我要 `core` 彻底无中文，以便守住"文字不入 core"红线、让 UI 自由做本地化与文案。
2. 作为玩家（经由未来 UI），下令成功后我要看到带数值的反馈（如农业变为 X(+Y)、民忠变为 X(+Y)、防灾变为 X(+Y)），这些数值由 core 事件携带。
3. 作为玩家，月末结算后我要看到结果（搜寻找到谁/什么、招降成败、掠得多少金粮、输送送达还是被劫、外交成败与势力变更），由 core 月末产出对应事件。
4. 作为玩家，我要看到月末系统事件（城市出现灾害、君主遭劫、请拥立新君、新君主产生、被策反成君、势力灭亡、势力归降），由 core 在对应结算点产出事件。
5. 作为玩家，同一类失败有多条台词、同一成功有多句随机台词时，由 UI 随机挑一句——core 不参与、**不消耗 `GameState.rng`**，现有确定性与随机序列不受扰动。

## 范围 & 不做

- 做：
  - 把所有 `canX` 的 `CommandCheck.reason` 从中文串改为**无中文的 code**（TS 字符串字面量联合，如 `'gold-insufficient'`）。同步更新引用 `reason` 的测试断言。
  - 为**经营类**命令（开垦/招商/搜寻/治理/出巡/招降/处斩/流放/赏赐/没收/交易/宴请/输送/移动/侦察/征兵/分配/掠夺，以及出征的"下令成功/部队已出发"）产出结构化结果事件（判别式 `kind` + 数值/id payload，无中文）。
  - 为**外交类**命令（离间/招揽/策反/劝降）产出结算结果事件（成败 + 目标 + 势力变更）。
  - 为**月末系统事件**（灾害出现、君主遭劫、待拥立新君、新君产生、被策反成君、势力灭亡、势力归降）产出事件。
  - 验收以 `docs/business-command-rules.md` 的经营/外交/系统类提示为对照清单：core 产出的 code/事件需能覆盖这些提示所需的判别与数值。
- 不做：
  - **战斗逐日提示**（地图战每日的进攻/胜败/俘虏/战死等交互式子对局内提示）不在本切片，留待后续。
  - **UI/store 层**：本切片只在 core 产出，不建界面、不做 code→中文 映射表的渲染（那是 UI 切片的事）。
  - **中文台词的具体挑选**：多变体台词由 UI 随机选，core 不挑、不存中文、不耗 RNG。
  - **失败校验本身的搬迁**：归属校验（"我方城池"等按己方判定）仍属 store 派发口，本切片不新建 store。
  - 不改任何既有游戏规则、数值公式、RNG 消耗序列。

## 验收标准

- [ ] `src/core/` 全树搜索无中文字符（含 `CommandCheck.reason` 与所有新增事件）。
- [ ] `CommandCheck.reason` 为字符串字面量联合 code；现有命令的失败原因一一映射到 code，相关测试改断 code。
- [ ] 经营/外交命令执行后可取得结构化结果事件，事件携带 UI 复现 `docs/business-command-rules.md` 对应提示所需的数值/id（如开发度新值+增量、掠得金/粮、搜寻所得类型与目标、招降/外交成败与目标、灾害城与灾种、遭劫/新君/灭亡/归降的相关 id）。
- [ ] 产出事件**不消耗 `GameState.rng`**；引入前后同 seed 的对局推进结果（state/RNG 序列）逐字节一致——现有 core 测试全绿、无需调随机期望值。
- [ ] 战斗逐日内的提示不在产出范围（仅出征"下令/部队已出发"在范围）。

## 待定决策

- **事件出口形态**（dev.md 定）：`apply`/`endMonth` 改为返回 `{ state, events }`，还是事件追加到 `GameState` 的一个字段（如 `outcomes`）由调用方读取后清空？前者改入口签名（改动放大大），后者引入新状态字段。需在 spec-dev 权衡，倾向选改动放大最小且不污染存档语义的方案。
- **AI 命令是否产事件**：core 多为 actor-agnostic，但文档明确"电脑势力通常不向玩家展示"。是 core 无差别产出、由 UI 按玩家相关性过滤，还是 core 对 AI 作弊简化路径不产事件？注意 AI 即时效果（`15`/`16`）部分不走玩家执行器，需在 dev.md 确认产出点。
- **自然恢复正常是否产事件**：文档称灾害自然恢复"不一定有独立提示"，是否需要 `city-recovered` 事件待定。
- **"电脑策略中……"进度提示**：纯 UI 进度态、非状态变更结果，倾向不进 core 事件（UI 自行展示），待确认。
